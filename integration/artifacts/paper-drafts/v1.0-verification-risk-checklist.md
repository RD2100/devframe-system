# v1.0 Verification Risk Checklist

## Scope

This checklist defines what a v1.0 verifier should cover for the local paper RAG human-review handoff. It is a checklist only; it does not implement verifier code.

Source materials reviewed:

- `CURRENT_DELIVERY.md`
- `scripts/verify_local_paper_rag_v0_9_handoff.py`
- `integration/artifacts/paper-drafts/local-paper-rag-clean-manuscript-v0.9.md`

Hard boundaries for the verifier:

- Do not read raw PDFs.
- Do not read raw Obsidian note bodies.
- Do not read secrets, keys, browser/CDP/cloud/MiniApp runtime content, external RAG services, cloud vector DB artifacts, raw chunks, vectors, FAISS binaries, WriteLab payloads, or Zotero keys.
- Do not treat a tool-chain check as final paper-quality acceptance, training-effect acceptance, production readiness, broad RAG readiness, or RuntimeAuthorization.

## P0 Checks

### 1. Required Artifact Existence

The verifier should fail if any required reviewer-facing artifact is missing.

Required artifact classes:

- DOCX manuscript.
- Markdown manuscript copy.
- Handoff ZIP package.
- Current delivery or release manifest that names the active artifacts.
- Verifier report output directory, if the verifier is expected to emit reports.

Risk covered: missing deliverable, stale pointer, or fake-green report over absent files.

### 2. Hash Integrity

The verifier should compute SHA256 for every committed handoff artifact and compare it with the expected hash declared by the v1.0 delivery manifest.

Minimum hash coverage:

- DOCX manuscript.
- Markdown manuscript copy.
- Handoff ZIP package.
- Any additional v1.0 evidence package that is explicitly part of the handoff.

Expected behavior:

- Hash mismatch is P0 fail.
- Missing expected hash is P0 fail unless the artifact is explicitly marked non-deterministic and excluded from the locked handoff.
- Report actual and expected hashes in machine-readable output.

Risk covered: artifact drift after human review, mismatched DOCX/Markdown pair, or package replacement.

### 3. ZIP Entries and Package Safety

The verifier should open the handoff ZIP and compare entries against the exact expected list from the v1.0 delivery manifest.

Minimum checks:

- Exact entry set; no missing files.
- No extra files.
- No absolute paths.
- No `..` traversal segments.
- No duplicate logical entries after path normalization.
- Expected manuscript DOCX and Markdown are present inside the package.
- Packaged file hashes match the standalone artifacts when those standalone artifacts are also part of the delivery.

Risk covered: packaging drift, hidden internal evidence leakage, unsafe paths, or stale manuscript copy inside the ZIP.

### 4. DOCX OpenXML Readability

The verifier should treat the DOCX as a ZIP and read at least `word/document.xml`.

Minimum checks:

- DOCX can be opened with `zipfile` or equivalent.
- `word/document.xml` exists and is UTF-8/XML readable.
- Document XML contains expected manuscript anchors, including DOI signals and boundary language.
- The DOCX does not require external network access to inspect its text XML.

Risk covered: corrupt DOCX, non-DOCX file with `.docx` extension, or DOCX that does not contain the expected manuscript text.

### 5. Markdown Manuscript Key Sentence Anchors

The verifier should fail if core v1.0 manuscript anchors disappear. Exact anchor strings should come from the v1.0 approved manuscript or delivery manifest.

For the v0.9-to-v1.0 baseline, preserve equivalent meaning for these anchors:

- Title frames the paper around virtual training systems, auxiliary value, scenario construction, and evaluation boundary.
- Abstract states that virtual training is an auxiliary/enhancement method, not a substitute for real training or real equipment operation.
- Abstract states that current evidence is mainly literature analysis and does not claim training effectiveness has been fully proven.
- Introduction asks how virtual training can provide bounded training support outside real training, rather than whether it can replace real training.
- Main contribution states three tasks: auxiliary training value, scenario-design points, and evaluation boundary.
- Scenario-construction section states the standard should shift from whether the scene "looks real" to whether it supports training tasks.
- Evaluation-boundary section states current material supports cautious auxiliary-training value, not a strong claim that training effectiveness is fully proven.
- Conclusion states virtual training should supplement and enhance real training systems, not replace real training, real equipment operation, or final capability evaluation.

Risk covered: accidental deletion of caution language or transformation into an overclaimed paper.

### 6. Forbidden Strong Claims

The verifier should fail on unqualified strong claims that exceed the manuscript's evidence boundary.

Strong-claim terms to detect as P0 unless they appear only inside an explicit "do not use" or negated boundary sentence:

- `显著提升`
- `有效替代`
- `全面改善`
- `取代真实训练`
- `替代真实训练`
- `训练效果已经得到充分证明`
- `真实任务能力已经提升`
- `岗位能力提升`
- `最终能力评价已经完成`

Expected behavior:

- The verifier should not blindly fail on every occurrence. The v0.9 manuscript intentionally includes examples of forbidden terms inside a boundary sentence: `而不是使用“显著提升、有效替代、全面改善”等强判断。`
- A v1.0 verifier should either whitelist the known boundary sentence or classify occurrences by nearby negation and boundary context.

Risk covered: converting a human-review handoff into unsupported empirical or operational acceptance.

### 7. Citation and Reference Basic Consistency

The verifier should fail if inline numeric citations and reference entries are structurally inconsistent.

Minimum checks:

- Inline citations use numeric bracket form such as `[1]`.
- Every cited number has a corresponding reference entry.
- Every required reference entry is cited at least once, unless explicitly allowed by the v1.0 manifest.
- Reference numbering is contiguous from `1`.
- No internal source markers such as `[S1]`, `[S2]`, or source-claim matrix artifacts appear in the public manuscript.
- DOI/article-number signals expected by the approved manuscript are still present.

Baseline v0.9 signals to preserve or intentionally replace in v1.0:

- Six reference entries.
- DOI `10.19384/j.cnki.cn11-5524/p.2018.04.003`.
- DOI `10.16338/j.issn.1009-1319.2013.06.003`.
- Article number `1009-0029(2010)11-0996-03`.
- Article number `2096-1227(2024)12-0046-03`.

Risk covered: broken citation numbering, leaked internal source markers, or public reference list drifting from cited claims.

### 8. Boundary Statement Preservation

The verifier should fail if the delivery boundary is missing or weakened.

Required boundary concepts:

- The handoff is for human review and is non-final.
- It is not final paper-quality acceptance.
- It is not training-effect acceptance.
- It is not production readiness.
- It is not broad RAG readiness.
- It is not RuntimeAuthorization.
- It does not authorize new live resource access.
- It does not inspect or include raw PDFs, raw Obsidian note bodies, raw chunks, source paths, vectors, FAISS binaries, WriteLab payloads, Zotero keys, browser/CDP/cloud/MiniApp runtime content, external RAG services, or cloud vector DB artifacts.

Risk covered: verifier output being misread as final approval or authorization for broader runtime/data access.

### 9. Fake-Green Guard

The verifier should fail closed.

Minimum checks:

- If any required path is missing, stop or mark final verdict `FAIL`.
- If any P0 check fails, final verdict must be `FAIL`.
- The report summary must include passed and failed counts.
- Failed or skipped checks must not be counted as passed.
- A generated report must not claim `PASS` if report writing failed.

Risk covered: failed command or partial evidence being reported as successful handoff verification.

## P1 Checks

### 1. DOCX and Markdown Text Drift

The verifier should compare normalized manuscript text extracted from DOCX OpenXML against the Markdown manuscript.

Recommended checks:

- Title appears in both.
- Section headings appear in both.
- Reference numbers and reference count match.
- Key boundary sentences appear in both.

Risk covered: reviewer-facing DOCX drifting from the Markdown source while both artifacts still exist.

### 2. Reference Metadata Completeness

The verifier should warn if reference entries lack basic bibliographic fields expected by the manuscript style.

Recommended fields:

- Author/title signal.
- Publication/source signal.
- Year.
- Journal, dissertation, or article-number/DOI signal when available.

Risk covered: structurally present but review-hostile reference entries.

### 3. Delivery Manifest Alignment

The verifier should ensure `CURRENT_DELIVERY.md` or the v1.0 equivalent points to the same active DOCX, Markdown, and ZIP artifacts that the verifier checks.

Recommended checks:

- Active artifact paths are named in the manifest.
- Manifest hashes match verifier expected hashes.
- One-command check and expected output are updated to v1.0 names.
- Supporting report paths, if present, use v1.0 names and dates.

Risk covered: verifier validates one set of files while reviewers are directed to another.

### 4. Report Determinism and Evidence Hygiene

The verifier should prefer deterministic report content unless timestamps are explicitly requested.

Recommended checks:

- Default output avoids wall-clock timestamps that churn committed reports.
- Reports list checked paths, expected hashes, actual hashes, and failed checks.
- Reports do not include raw source snippets from prohibited inputs.
- Reports preserve enough evidence for reviewer verification without exposing forbidden content.

Risk covered: noisy diffs, stale evidence, or accidental leakage in generated reports.

### 5. Human-Review Reminders

The verifier should include non-blocking reminders for items that automation cannot prove.

Recommended reminders:

- Human reviewer must decide intended use case: short paper, technical note, review-style course paper, or internal research brief.
- Human reviewer must judge whether the formalized title, abstract, contribution statement, and conclusion match the intended use case.
- Human reviewer must judge whether citation style is acceptable for the target venue.
- Human reviewer must decide whether stronger empirical training-effect evidence is required.

Risk covered: automation overclaiming scholarly judgment or venue acceptance.

## Suggested P0/P1 Split

P0 should block handoff when:

- Required files are missing.
- Hashes do not match the delivery manifest.
- ZIP entries are missing, extra, unsafe, or stale.
- DOCX OpenXML cannot be read.
- Public manuscript loses required caution/boundary anchors.
- Unsupported strong claims appear outside allowed negated/boundary contexts.
- Citation numbers and reference entries are inconsistent.
- Boundary statements are missing or weakened.
- The verifier would otherwise report pass despite failed checks.

P1 should warn, not block, when:

- DOCX and Markdown have minor normalized text drift that does not remove key anchors.
- Reference metadata is present but stylistically incomplete.
- Manifest/report wording is stale but artifacts and hashes are still unambiguous.
- Deterministic reporting or evidence hygiene can be improved.
- A human-review judgment remains necessary.

## Reviewer Focus

Reviewers should pay closest attention to:

- Whether v1.0 expected hashes are sourced from a reviewed manifest rather than copied blindly from v0.9.
- Whether forbidden-claim detection handles negated examples without false P0 failures.
- Whether citation/reference checks catch missing entries without reading original PDFs or Obsidian notes.
- Whether DOCX extraction checks actual OpenXML text and not only ZIP readability.
- Whether final verifier verdicts remain non-final and human-review scoped.
